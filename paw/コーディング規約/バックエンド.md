# バックエンドコーディング規約

## 技術スタック

- Java / Spring Boot
- Spring Data JPA
- Lombok
- Gradle マルチプロジェクト構成

## モジュール構成と依存関係

```
api ──→ domain, external, support
batch ──→ domain, external, support
domain ──→ support
external ──→ support
support（依存なし）
```

モジュール間の依存方向は上記の通りに限定する。**ドメイン同士は直接参照しない**。ドメインをまたぐ参照はIDのみを使う。

## パッケージ命名

```
com.konekokonekone.nekodion.<ドメイン名>.<レイヤー名>
```

例：

- `com.konekokonekone.nekodion.transaction.entity`
- `com.konekokonekone.nekodion.transaction.service`
- `com.konekokonekone.nekodion.api.controller`

## レイヤー構成

### api モジュール

| パッケージ   | 責務                                           |
| ------------ | ---------------------------------------------- |
| `controller` | HTTPリクエストの受付・レスポンスの返却         |
| `usecase`    | APIユースケースの組み立て（Serviceの呼び出し） |
| `request`    | リクエストボディのモデル                       |
| `response`   | レスポンスボディのモデル                       |
| `mapper`     | DTO/Entity → Response の変換                   |
| `security`   | 認証・認可設定                                 |
| `config`     | Spring設定クラス                               |

### domain モジュール

| パッケージ   | 責務                                           |
| ------------ | ---------------------------------------------- |
| `entity`     | JPAエンティティ                                |
| `repository` | Spring Data JPAリポジトリ                      |
| `service`    | ドメインロジック                               |
| `dto`        | ドメイン間・レイヤー間のデータ転送オブジェクト |
| `enums`      | 列挙型                                         |

## 命名規則

| 種別             | 命名例                       |
| ---------------- | ---------------------------- |
| Controller       | `TransactionController`      |
| UseCase          | `TransactionUseCase`         |
| Service          | `TransactionService`         |
| Repository       | `TransactionRepository`      |
| Entity           | `Transaction`                |
| Request          | `TransactionRequest`         |
| Response（一覧） | `TransactionSummaryResponse` |
| Response（詳細） | `TransactionDetailResponse`  |
| DTO              | `TransactionRequestDto`      |
| Mapper           | `TransactionItemMapper`      |

一覧用と詳細用でレスポンスクラスが分かれる場合は `XXXSummary` / `XXXDetail` で使い分ける。一覧・詳細が対になっていない場合はサフィックスを省略して `XXXResponse` とする。

## Controller の規約

- `@RestController` + `@RequestMapping("/api/<リソース名>")` を付ける
- `@RequiredArgsConstructor` でコンストラクタインジェクション
- ビジネスロジックは持たず、UseCase に委譲する
- 認証ユーザーは `@CurrentUser UserDto currentUser` で受け取る（`SecurityContext` に直接アクセスしない）
- HTTPステータスは意味に応じて使い分ける（200 OK / 201 Created / 204 No Content）

```java
@GetMapping("/{id}")
public ResponseEntity<TransactionDetailResponse> getTransaction(
        @CurrentUser UserDto currentUser,
        @PathVariable Long id) {
    return ResponseEntity.ok(transactionUseCase.getTransaction(id, currentUser.getId()));
}
```

## ID 生成戦略

URLのパスに露出するIDは連番（AUTO INCREMENT）を使わない。連番は総件数の推測や他ユーザーのリソースへのアクセス試行を許すため、ユーザーがURLで直接アクセスするリソースには `IdGenerator.generate()` による独自IDを使う。

| 用途                      | ID種別                   | 例             |
| ------------------------- | ------------------------ | -------------- |
| URLパスに含まれるリソース | `IdGenerator.generate()` | 取引履歴、口座 |
| 内部管理・ログ等          | AUTO INCREMENT           | ログテーブル   |

## Entity の規約

- `AbstractBaseEntity` を継承する
- URLパスに含まれるリソースのIDは `@PrePersist` で `IdGenerator.generate()` により生成する
- 内部管理用テーブル（ログ等）のIDは AUTO INCREMENT を使う
- Setter/Getter は Lombok `@Getter` `@Setter` を使う
- 列挙型カラムには `@Convert` でカスタムコンバーターを使う
- 遅延ロードが必要な関連は `fetch = FetchType.LAZY` を明示する

## Repository の規約

- `JpaRepository<Entity, Long>` を継承する
- 複雑なクエリは `@Query` + JPQL テキストブロックで記述する
- JPQL は SELECT / FROM / WHERE / ORDER BY を改行して揃える

```java
@Query("""
        SELECT
            t
        FROM
            Transaction t
        WHERE
            t.userId = :userId
        ORDER BY
            t.transactionDateTime DESC
        """)
List<Transaction> findByUserId(String userId);
```

## ユーザー認証

- コントローラーのメソッド引数に `@CurrentUser UserDto currentUser` を付ける
- サービス・リポジトリはユーザーIDを `String userId` として受け取る
- `SecurityContextHolder` を直接参照しない

## Lombok

- フィールドインジェクションは使わない。`@RequiredArgsConstructor` + `final` フィールドで統一する
- データクラスには `@Getter` `@Setter` のみを付ける（`@Data` は使用しない）

## ドメイン間依存の禁止

異なるドメイン（例：`transaction` と `category`）を直接参照しない。参照が必要な場合はIDのみを使い、ルックアップはユースケース層で行う。

## 命名・コーディングスタイル

- メソッド名はキャメルケース
- クラス名はパスカルケース
- 定数は `UPPER_SNAKE_CASE`
- コメントはJPQL等の複雑なクエリに限定し、自明なコードにはつけない
- ローカル変数の型は `var` を使い、import を最小限に抑える

```java
// Good
var transactions = transactionRepository.findByUserId(userId);
var response = transactionMapper.toResponse(transactions);

// Bad
List<Transaction> transactions = transactionRepository.findByUserId(userId);
TransactionDetailResponse response = transactionMapper.toResponse(transactions);
```
